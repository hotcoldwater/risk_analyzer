# Audit Risk Intelligence

> **산업의 경제적 실질과 재무제표 숫자를 함께 읽어, 감사·재무 리스크를 설명하는 한국 상장기업 분석 워크벤치**

`Audit Risk Intelligence`(이하 ARI)는 단일 재무비율을 조회하는 서비스가 아니다. 산업별 사업구조와 회계 쟁점을 전제로, 기업의 재무 흐름을 동종기업과 비교하고 **왜 확인이 필요한지**, **어느 계정·공시를 더 봐야 하는지**를 근거와 함께 제시하는 분석 서비스다.

현재는 방산 기업의 현금화 리스크 분석이 동작하는 첫 번째 vertical slice다. `look/`의 최신 번들은 방산·반도체·건설로 확장할 수 있는 출발점이며, 제품은 “방산 전용 대시보드”가 아니라 **산업별 감사 리스크 엔진**으로 재정의한다.

## 제품 방향

### 우리가 푸는 문제

회계법인·감사인·재무분석 담당자는 숫자를 찾는 데서 끝나지 않는다. 산업에 맞지 않는 비교군, 연결/별도 혼용, 수익인식 방식의 차이, 계정명 차이 때문에 숫자의 의미를 다시 해석해야 한다. ARI는 다음 작업을 하나의 흐름으로 연결한다.

```text
기업/산업 선택
  → 비교군과 재무제표 기준 확정
  → 산업별 핵심 위험 신호 계산
  → 시계열·동종분포·원천계정으로 검증
  → 감사 관점의 가설·확인 질문·근거를 리포트로 정리
```

### 핵심 사용자와 결과물

| 사용자 | 필요한 답 | ARI가 제공할 결과 |
| --- | --- | --- |
| 감사 실무자·매니저 | 어디를 우선 검토해야 하는가? | 위험 신호, 계정별 변화, 감사 확인 질문, 근거 링크 |
| 재무·IR 담당자 | 동종사 대비 우리 위치와 설명 포인트는 무엇인가? | 산업 벤치마크, 추이, 차이의 원인 후보 |
| 리서치·투자 분석가 | 숫자의 질이 사업 맥락과 맞는가? | 산업 드라이버, 이상치, 공시 확인 경로 |

### 제품 원칙

1. **점수보다 근거** — 종합 등급은 탐색의 시작일 뿐이며, 모든 신호는 산식·기간·표본·원천 계정으로 역추적 가능해야 한다.
2. **산업 우선** — 전 산업 공통 비율만으로 결론을 내리지 않는다. 산업별 위험 라이브러리와 비교군 정책이 먼저다.
3. **데이터 기준을 숨기지 않기** — 연결/별도, 회계연도, 표본 수, 결측과 제외 사유를 항상 보인다.
4. **판단을 보조하되 자동 판정하지 않기** — “고위험”은 감사 의견이나 부실 판정이 아니라 추가 검토 우선순위다.
5. **재현 가능성** — 같은 데이터 버전과 규칙 버전이면 같은 결과가 재현되어야 한다.

### UX 전환 — 분석 도구형 밀도의 단일 홈 테이블 (반영 완료)

이전에는 큰 헤드라인·여러 안내 카드·사이드바가 먼저 보여 서비스 소개 화면처럼 느껴졌고, "기업 분석"과 "산업 분석"이 별도 화면으로 분리돼 있었다. 지금은 이를 단일 홈 화면으로 통합했다: **홈 화면 = 산업 탭 전환형 전체 기업 테이블**이며, 별도의 산업 랜딩·산업 인사이트 페이지/모달은 없앴다. 사용자 피드백("산업/기업 화면을 나누지 말고 바로 전체 기업을 보여달라", "산업분석 모달 자체를 없애자")을 반영한 결과다.

| 구분 | 이전 | 현재 |
| --- | --- | --- |
| 첫 화면 | 산업 카드 그리드 → 산업 선택 후 비교 테이블 | 진입 즉시 전체 기업 테이블. 산업은 드롭다운이 아니라 나열된 탭 버튼(방산/반도체/건설)으로 전환 |
| 기업 탐색 | 카드·화면 전환 중심 | 좌측 정렬/검색/검토신호 필터 가능한 기업 테이블, 선택 상태를 유지 |
| 기업 분석 | 별도 워크스페이스의 큰 섹션 | 우측 고정 인스펙터에서 기존 리스크 분석·계정 추이·감사 질문·메모를 즉시 표시 |
| 산업 분석 | 산업 카드와 별도 비교/인사이트 화면·모달 | **제거함.** 지표 열은 3개 산업 공통(매출액·영업이익률·영업현금흐름 전환율)으로만 표시하고, 산업별 세부 지표는 인스펙터에서 확인 |
| 시각 언어 | 큰 타이포그래피·카드·강한 여백 | 작은 제목, 표 중심, 절제된 구분선·배지·탭, 숫자 정렬과 정보 밀도 우선 |

**레이아웃 원칙:** 데스크톱에서는 `좌측 기업 테이블 : 우측 기업 인스펙터 = 약 60 : 40`의 2단 구조를 기본으로 한다. 행 선택은 페이지 이동 없이 우측만 갱신한다. 모바일에서는 인스펙터를 하단 시트로 전환한다.

**우측 인스펙터의 고정 구성:** 선택 행은 표에서 강조한다. 우측에는 `(1) 기업명·종목코드·기준연도`, `(2) 산업별 핵심 계정 4~5개의 3개년 스파크라인`, `(3) 공통 지표 3개(매출액·영업이익률·영업현금흐름 전환율)의 조밀한 표`, `(4) ARI 고유의 위험 신호·근거 계정·감사 확인 질문·검토 메모`를 순서대로 표시한다. `전체 분석 열기`는 기존 URL 기반 심화 워크스페이스(방산 탭 구조 포함)로 이어진다.

**색과 규칙 원칙:** 초록·노랑·빨강 배지는 빠른 스캔을 돕는 보조 수단이다. 모든 색상 신호에는 규칙명·산식·표본 수·CFS/OFS·제한사항을 연결한다. 임계값을 시각적으로만 단정하지 않는다.

**알려진 제한사항:** 방산의 위험 신호·검토 신호는 아직 이 통합 테이블에서 "판단 전"으로만 표시된다 (계약자산 기반 정식 신호는 기업 상세의 현금화 리스크 분석에서만 제공). CFS/OFS 기준을 계정 하나가 아니라 회사 단위로 한 번에 선택하는 현재 로직 때문에, DB에 값이 있어도 다른 기준이 선택되면 화면에는 N/A로 보이는 경우가 있다 — 이는 DART 재추출로 해결되는 문제가 아니라 별도의 기준 선택 로직 개선 과제로 분리해 관리한다.

### 지금 테스트 가능한 범위

- 최신 번들 검증: `--default-updated-at 2026-07-11 --auto-map-unclassified-industries --reconcile-stock-codes --validate-only` 옵션으로 DB 변경 없이 전체 번들을 검증한다.
- 홈 화면: 앱 진입 시 바로 전체 기업 테이블이 열린다. 상단 산업 탭(방산/반도체/건설)으로 전환하고, 연도(2021–2025, 3산업 공통)·검색·검토신호 필터를 조합해 확인한다. 증가율·수익성 지표는 양수(초록)/음수(빨강)로 구분돼 보인다.
- 방산 UI: 유효한 Supabase 연결 상태에서 기업을 검색하거나 테이블 행에서 `전체 분석 열기`를 누르면 `/companies/{corp_code}?industry=defense` 형태의 워크스페이스가 열리고 새로고침·뒤로가기로 컨텍스트가 복원된다.
- 반도체: 35개사 비교·정렬·위험 신호·기업 상세가 운영 DB 기준으로 동작하며 2021–2025 5개년 데이터를 확보했다(현재 UI는 2025년 기본 조회). 원문 DART 링크는 접수번호를 데이터 계약에 추가한 뒤 연결한다.
- 건설: 65개사를 안전 적재했으며, 영업이익·계약자산(미청구공사) 등 주요 결측을 DART 재추출로 대부분 해소하고 2021–2025 5개년으로 확장했다. 비교군 정책과 산업 규칙을 설계하는 단계다.
- 데이터 결측 진단: `backend/.venv/bin/python scripts/supabase/diagnose_data_gaps.py`로 DB의 결측 셀을 원인별(행 자체 없음/값만 없음)로 즉시 재확인할 수 있다 (현재는 기존 산업별 핵심 계정만 검사 대상이며, 신규 표준 재무비율 14계정은 아직 이 진단 스크립트 범위에 없다).

### 사용자 테스트 체크포인트 — Phase 1 완료

Phase 1의 방산 흐름은 지금 테스트하는 것이 가장 좋다. 로컬 프론트/백엔드와 방산 데이터가 있는 Supabase 연결을 실행한 뒤 아래를 확인한다.

1. 방산 기업을 검색하고, 검색 결과에서 기업·산업을 선택한다.
2. 주소가 `/companies/{corp_code}?industry=defense`로 바뀌고 새로고침 뒤에도 같은 기업이 열리는지 확인한다.
3. 현금화 리스크 지표를 바꾸며 현재값, 동종 평균, 백분위·사분위, 원천 계정 상세가 자연스럽게 이해되는지 확인한다.
4. 검토 메모를 입력하고 새로고침해도 같은 브라우저에 남는지, `분석 링크 복사`와 `인쇄용 리포트`가 기대대로 동작하는지 확인한다.

메모는 현재 서버가 아니라 해당 브라우저의 localStorage에만 저장된다. 다른 기기·브라우저에 공유되는 메모 기능은 Phase 3/4의 권한·리포트 저장 모델 이후에 제공한다.

### 다음 사용자 테스트 체크포인트 — Phase 2 반도체 비교

로컬 백엔드와 프론트를 재시작한 뒤 홈 화면 상단 산업 탭에서 **반도체**를 선택한다. 35개 기업이 보이고, 공통 지표(매출액·영업이익률·영업현금흐름 전환율) 헤더를 눌러 정렬이 바뀌는지 확인한다. `주의`/`위험` 행을 눌러 우측 인스펙터에서 재고 부담·CAPEX 부담·현금흐름 전환 저하의 근거가 이해되는지 확인한다.

### 결측 데이터 진단·DART 재추출 보강 파이프라인

UI에 N/A로 보이는 셀은 원인이 섞여 있어 무작정 DART를 재호출하면 이미 DB에 있는 값까지 중복 수집하게 된다. 진단 결과 결측은 크게 두 유형이다: **①  source_gap** — 해당 corp_code/year/account 행이 CSV 번들에 아예 없음, **② null_amount** — 행은 있는데 값이 비어 있음. (③ 계정은 DB에 있지만 앱이 CFS/OFS 기준을 회사 단위로 하나만 선택해서 화면에는 안 보이는 경우도 있는데, 이는 DART 재추출과 무관한 별도 이슈로 분리해 관리한다.)

- `scripts/supabase/diagnose_data_gaps.py`: `companies_basic × industry_map × 산업별 재무 사실 테이블 × 필수 계정 × 실제 존재 연도`를 대조해 결측 셀을 산업·기업·계정별로 집계하고 원인을 분류한다. 읽기 전용이며 DB를 바꾸지 않는다.
- `scripts/dart/main.py`: 기존 DART 추출기. 결측이 확인된 corp_code 목록으로 전체 재무제표(`fnlttSinglAcntAll`)를 재수집해 로컬 SQLite에 적재한다.
- `scripts/dart/fill_missing_accounts.py`: SQLite에서 특정 표준 계정 하나를 골라 DART 원문의 계정명 변형(예: "영업이익(손실)", "IV.영업이익")을 표준명으로 매핑하고, 이미 CSV에 있는 자연키와 겹치지 않는 행만 번들에 추가한다. `--dry-run`으로 먼저 규모를 확인할 수 있다.

**2026-07-17 적용 사례:** 건설 `영업이익`이 65개사 × 3개년(2023–2025) 전체에서 결측이었다(원인: 반도체는 CSV에 `당기순이익`이 있었지만 건설은 애초에 `영업이익` 행 자체가 번들에 없었음). 위 도구로 DART에서 360행을 재추출·매핑·병합해 `--validate-only` 통과를 확인한 뒤 실제 Supabase에 반영했다. 반영 후 재진단 결과 건설 결측은 346건 → 151건으로 줄었고(계약자산·충당부채·차입금·매출채권만 남음), API 응답에서 건설 65개사 전부 영업이익률이 계산되는 것을 확인했다.

**2026-07-18 적용 사례 — 계약자산(미청구공사)과 "N/A ≠ 결측"의 구분:** 건설 `계약자산(미청구공사)`은 회사마다 원문 라벨이 "단기미청구공사", "유동계약자산" 등으로 갈려 있어 우선순위 매핑(동일 계정명 중복 시 statement `ord` 최솟값 채택, 다른 변형이 섞이면 값이 있는 후보를 우선하고 그다음 호출자가 지정한 우선순위를 따름)을 `fill_missing_accounts.py`에 추가해 198건을 더 채웠다(151건 → 75건). 방산 27개사 "전체 결측"은 DB의 `memo` 필드를 직접 열어보고서야 대부분이 버그가 아님을 확인했다: **"해당 연도 원천 재무제표 없음"**(최근 상장이라 그 이전 연도 재무제표 자체가 없음)과 **"해당 연도 재무제표에 해당 계정 없음"**(그 해에 실제로 그 계정이 없음)은 정상이며 DART 재추출로 채워지지 않는다. 자동 매칭이 남긴 **"표준 매칭 없음, 유사계정: X"** 힌트도 절반 가량은 계약자산(자산)에 계약부채(부채) 계열을 잘못 추천한 오탐이었다 — DART 원문을 직접 대조해 개념이 맞는 것만(AP위성·빛샘전자의 `단기미청구공사`/`단기초과청구공사`, 풍산의 `영업활동순현금흐름`) 17건 반영했다. **교훈:** N/A 셀을 일괄 재추출 대상으로 보지 말고, 먼저 `memo`가 이미 원인을 설명하고 있는지 확인해야 한다.

**2026-07-18 대규모 확장 — 3개 산업 147개사 5개년·표준계정 통일:** 방산·반도체·건설 전체(47+35+65=147개사)를 DART에서 2021–2025 5개년·CFS/OFS로 재추출하고, `fill_missing_accounts.py`로 표준 재무비율 계정 14개를 반복 반영했다(총 11,324행 추가 — 계정마다 `fill_missing_accounts.py`를 한 번씩 호출하는 반복 스크립트로 처리). 이 과정에서 `fill_missing_accounts.py`의 dedup 로직 결함을 하나 더 찾아 고쳤다: 반도체 CSV 일부 셀에 스프레드시트 내보내기 과정에서 생긴 `01199550"`처럼 따옴표가 값에 섞여 들어간 원본이 있었는데(README의 "반도체 빈 헤더·따옴표·지수 표기" 이슈와 동일 계열), dedup 비교가 이를 정상 값과 다른 기업으로 오인해 중복 행을 만들 뻔했다 — 업로더의 `clean_identifier`와 동일하게 모든 키 필드에서 `strip('"')`를 적용하도록 고쳤다(corp_code 8자리 정규화는 계약자산 작업 때 이미 반영됨). 또한 `당기순이익`도 `영업이익`처럼 "당기순이익(손실)" 변형이 있어 방산·건설에 추가 반영했다. 최종 회사×연도 커버리지는 위 "표준 재무비율 계정 세트" 표를 참고.

"기본적인 재무비율 분석"이 항상 가능하려면 5개 축(수익성·안정성·유동성·활동성·성장성)에 필요한 계정이 3개 산업 전체에 있어야 한다. 2026-07-18에 아래 14개 표준 계정을 3개 산업 공통으로 DB에 채워 넣었다(산업별 특화 계정인 계약자산/계약부채/차입금/충당부채/CAPEX 등은 별도로 유지).

`이자비용`은 대부분의 기업이 재무제표 본문에 별도 캡션으로 공시하지 않고 주석에만 두기 때문에 커버리지가 낮다 — 추출 실패가 아니라 공시 관행의 한계이며, 이자보상배율은 당분간 일부 기업만 계산 가능하다.

**UI 반영 완료 (2026-07-18):** 산업 비교 표에 매출총이익률·순이익률·부채비율·유동비율·총자산회전율·ROE 6개 열을 추가했다(공통 지표: 매출액·영업이익률 + 위 6개 + 영업현금흐름 전환율, 총 9열). `backend/app/defense_service.py`의 `INDUSTRY_COMPARISON_ACCOUNTS`를 `STANDARD_RATIO_ACCOUNTS` 기준으로 3개 산업 공통 재정의하고, `_comparison_metrics`에 각 비율 산식을 추가했다. 우측 인스펙터의 "현재값과 보조 비율" 표에도 자동으로 함께 표시된다.

**남은 과제:**
1. 활동성 비율(재고자산회전율·매출채권회전율)은 아직 테이블에 없다 — 필요성 논의 후 추가할 수 있다.
2. `이자비용`은 `금융비용`(더 넓은 개념, 이자 외 비용 포함)으로 대체 계산할지 여부를 결정해야 한다. 이자보상배율은 이번 라운드에서 열로 추가하지 않았다(커버리지 3~8%로 대부분 N/A가 되기 때문).
3. CFS/OFS 기준을 회사 단위로 하나만 선택하는 현재 로직(위 "알려진 제한사항" 참고)은 이번 확장과 별개로 여전히 남아 있다 — 데이터는 CFS/OFS 모두 저장돼 있지만 조회 시 한쪽만 보인다.

### 공통 분석 코어

모든 산업에 공통 적용하는 화면과 계산 단위다.

- 기업 카드: 시장, KSIC, 산업 분류, 수익인식 기준, 데이터 기준일
- 재무 흐름: 매출·영업이익·영업현금흐름·자산·부채·자본의 시계열
- 비교: 지정 산업·그룹(A / A+B / A+B+C) 기준의 중앙값, 사분위, 표본 수
- 이상 신호: 전년 대비 급변, 이익과 현금흐름 괴리, 동종분포 이탈, 결측/기준 변경
- 증거: 계정명, 금액, 재무제표 구분, 산식, 데이터 버전, 원천 메모
- 증가율·수익성 지표는 부호(+/−)에 따라 색상을 구분해 표시한다(양수 초록, 음수 빨강). 기업 비교 표·인스펙터·방산 지표 카드에 적용했다.

| 반도체 | 재고평가·CAPEX·수익성 사이클 | 재고/매출, 재고 회전일수, 매출총이익률, EBITDA, CAPEX/매출, 영업현금흐름 | 재고와 투자 증가가 수요/가동률 변화와 일관되는가? 비용·감가상각 부담은 감당 가능한가? |
| 건설 | 진행기준 수익·미청구공사·원가/충당 | 계약자산(미청구공사)/매출, 채권 회수, 원가율, 충당부채, 차입금, 영업현금흐름 | 공정률과 청구·회수가 일치하는가? 손실 프로젝트와 우발부채 징후는 없는가? |

산업 모듈은 다음 5개 요소를 한 묶음으로 등록한다: `비교군 정의`, `필수 계정`, `지표 산식`, `위험 규칙/임계값`, `감사 확인 질문`. 임계값은 일괄 하드코딩하지 않고 산업·규모·그룹별 설정과 규칙 버전으로 관리한다.

## 목표 UX/UI

검색 결과는 URL 기반 **분석 워크스페이스**로 전환됐다. 검색은 진입점일 뿐, 결과는 주소를 공유하고 새로고침해도 복원되는 독립 페이지여야 한다.

```text
좌측: 산업/기업 탐색 · 저장 목록
상단: 기업명 · 기준연도 · CFS/OFS · 비교군 · 데이터 기준일
본문: [리스크 개요] [테마별 분석] [동종 비교] [원천 데이터] [리포트]
우측: 활성 위험 신호 · 확인 질문 · 주석/내보내기
```

### 화면 우선순위

1. **기업 분석 2단 워크스페이스** — 산업·연도·검색·위험 필터와 KPI를 압축 배치하고, 좌측 기업 표와 우측 상세 인스펙터를 한 화면에 둔다.
2. **우측 기업 인스펙터** — 선택 행 강조, 핵심 지표 스파크라인, 조밀한 비율 표와 기존 위험 신호·계정 추이·동종 분포·감사 질문·검토 메모를 선택 기업에 맞춰 즉시 갱신한다.
3. **산업 분석 전용 페이지** — 산업 KPI, 위험 신호 분포, 그룹·연도별 비교, 데이터 커버리지와 제한사항을 시각화한다.
4. **리뷰·공유** — 검토 메모, 리포트 내보내기, 공유 링크, 데이터 기준일/규칙 버전을 제공한다.

### UI 원칙

- “고위험” 색만 보이지 않게 한다. 등급 옆에 **신호 수·변화량·근거 계정**을 항상 둔다.
- 평균 하나 대신 중앙값·사분위·표본 수를 기본으로 하고, 제외된 기업 수와 사유를 표시한다.
- `2025년`, `CFS`, `A 그룹` 같은 분석 컨텍스트는 화면 상단에 고정해 오해를 막는다.
- 숫자는 억/조 단위로 읽기 쉽게 보이되 hover/증거 탭에서 원 단위와 원천 계정을 확인할 수 있게 한다.
- 빈 상태는 “준비 중”으로 끝내지 않는다. 해당 산업의 커버리지, 필요한 데이터, 출시 예정 분석을 안내한다.
- 전문 서비스답게 절제된 밝은 바탕·블루 액센트를 유지하되, 색에 의존하지 않는 배지·아이콘·문장형 설명을 사용한다.

## 목표 아키텍처

```text
Source CSV / DART / 수동 검토
        ↓
Ingestion + schema validation + quality report
        ↓
Canonical financial facts (회사·연도·CFS/OFS·재무제표·표준계정)
        ↓
Industry membership + peer-group policy + accounting policy metadata
        ↓
Metric engine → risk-rule engine → evidence/report API
        ↓
React analysis workspace / export
```

### 데이터 모델 개편 원칙

- `companies`: 법인코드 중심의 단일 마스터. 종목코드, 시장, KSIC, 수익인식 메타데이터를 포함한다.
- `industry_memberships`: 기업이 여러 산업에 속할 수 있는 관계와 `is_primary`, 레벨, 편입 사유, 유효기간을 관리한다.
- `financial_facts`: `corp_code + year + fs_div + sj_div + canonical_account`를 기준으로 한 정규화 사실 테이블. 원천 계정명과 raw 값도 보존한다.
- `account_mapping`: 산업별 원천 계정명 ↔ 표준 계정의 매핑, 신뢰도, 검토 상태를 버전 관리한다. 예: `계약자산(미청구공사)` → 계약자산.
- `analysis_definitions` / `risk_rules`: 지표와 규칙을 코드에서 분리하고, 산식·필수 계정·설명·버전을 데이터로 관리한다.
- `data_runs` / `quality_issues`: 적재 단위의 manifest, 검증 결과, 행 수, 제외 사유, 승인 상태를 남긴다.

### API 재편 방향

현재 `/api/analysis/liquidity-metric`와 `/api/analysis/anomaly`는 방산/2025 중심이다. 목표 API는 산업·기간·재무제표 기준을 명시적으로 받는다.

- `GET /api/industries` — 산업별 커버리지와 최신 데이터 상태
- `GET /api/industries/{industry_id}/companies` — 필터·정렬 가능한 산업 비교 목록
- `GET /api/companies/{corp_code}/overview?industry_id=&year=&basis=` — 개요와 활성 신호
- `GET /api/companies/{corp_code}/themes/{theme_code}` — 시계열, 동종분포, 산식, 증거
- `GET /api/companies/{corp_code}/evidence` — 원천 계정·메모·데이터 버전
- `POST /api/reports` — 선택한 분석 컨텍스트와 메모를 고정한 리포트 생성

## 개발 로드맵

### Phase 0 — 데이터 계약 확정과 안전한 적재 (2주) · 운영 반영 완료

**목표:** `look/`의 최신 번들을 재현 가능하게 적재하고, 분석 가능한 범위와 불가능한 범위를 시스템이 판별한다.

- 새 CSV 스키마와 `manifest.json` 정의, 샘플 및 데이터 사전 작성
- 기업 마스터·산업맵 정합화: 건설 65개사 보강, 멀티 산업 소속 정책 확정
- 반도체 헤더/따옴표/중복 정제, 숫자 형식(콤마·지수표기) 파서 강화
- 표준 계정 매핑 초안: 특히 건설 `계약자산(미청구공사)`의 정규화
- 적재 전 검증(참조 무결성, 자연키 중복, 기간·CFS/OFS·필수계정 커버리지)과 품질 리포트 생성

**완료 기준:** 세 산업 번들이 검증을 통과하고, 각 산업의 기업 수·기간·결측·제외 목록이 데이터 품질 화면/파일로 재현된다.

**현재 결과:** 승인된 최신 번들은 Supabase에 적재됐다. 기업 144개, 산업맵 147개, 방산 1,645행·반도체 2,135행이며, 방산 기업 검색과 지표 계산을 실제 DB에서 검증했다. 건설은 초기 2,974행에서 DART 재추출로 영업이익 360행을 보강해 현재 3,334행이다 (아래 "결측 데이터 진단·DART 재추출 보강 파이프라인" 참고).

### Phase 1 — 방산 MVP를 “증거 있는 리스크 개요”로 완성 (3주) · 사용자 테스트 준비 완료

**목표:** 현재 동작하는 방산 현금화 분석을 실제 업무에 쓸 수 있는 기업 상세 페이지로 전환한다.

- 완료: 모달을 URL 기반 기업 상세 워크스페이스로 교체
- 완료: 현재값의 동종 백분위와 중앙값·사분위·최저/최고 분포, 산식·출처·제한사항·원천 계정 드릴다운을 연결
- 완료: 브라우저별 검토 메모, 현재 분석 URL 복사, 인쇄용 리포트 제공
- 현금화/수익인식 테마: 계약자산, 순계약자산, 영업현금흐름 전환율, 채권의 추이·동종분포·계정 근거
- 위험 등급에 규칙 버전, 산식, 표본 수, 결측 제한사항을 연결
- A/A+B/A+B+C 선택을 비교군 설명과 함께 제공; 중앙값·사분위 비교 도입
- 분석 결과 내보내기(초기 HTML/인쇄용)와 상태별 테스트 추가

**완료 기준:** 방산 47개사에서 동일한 컨텍스트로 재현 가능한 리스크 개요가 나오고, 모든 신호가 원천 계정과 규칙으로 드릴다운된다.

### Phase 2 — 산업 비교와 반도체 모듈 (3–4주) · 진행 중

**목표:** 기업 단건 조회를 산업 단위 탐색으로 확장한다.

- 산업 랜딩·산업 비교 테이블·스크리너(그룹, 시장, 위험도, 데이터 충족도)
- 반도체 모듈: 재고, 수익성, CAPEX/현금흐름, 무형자산 관련 리스크
- 비교 분포·백분위·동종사 선택 기능과 시계열 차트 표준화
- 지표 정의/규칙을 설정형 레지스트리로 이관

**완료 기준:** 반도체 35개사를 대상으로 최소 3개 테마를 비교·정렬·기업 상세로 연결할 수 있다.

**현재 완료 범위:** `GET /api/industries`와 산업 랜딩 화면이 실제 Supabase의 기업 수·분류 상태를 표시한다. 방산만 분석 가능으로 노출하며, 반도체·건설은 데이터 준비 상태로 명확히 구분한다.

**현재 완료 범위 추가:** `GET /api/industries/semiconductor/comparison?year=2025`는 35개사를 매출액·영업이익률·재고/매출·CAPEX/매출·영업현금흐름 전환율로 반환한다. UI에서는 각 열 정렬, 연결(CFS) 우선 기준, 필수 계정 충족도와 행 상세를 제공한다. 재고/매출·CAPEX/매출은 산업 3사분위 이상, 영업현금흐름 전환율은 0.5배 미만일 때 검토 신호로 표기한다. 이는 결론이 아니라 추가 확인 우선순위다.

**현재 완료 범위 추가:** 행 상세의 `기업 상세 열기`는 `/companies/{corp_code}?industry=semiconductor` 워크스페이스로 이동한다. 선택 기업의 2025년 비교 지표, 위험 신호, 연결/별도 기준, 필수 계정 충족도를 표시한다.

**현재 완료 범위 추가:** 반도체 상세는 매출액·재고자산·유형자산의 취득(CAPEX)·영업활동현금흐름의 3개년 원천 계정 흐름과 각 위험 신호에 맞는 감사 확인 질문을 제공한다. CFS 우선/OFS 보조와 임계값의 제한사항도 함께 명시하며, 방산과 동일하게 브라우저 검토 메모·분석 URL 복사·인쇄를 사용할 수 있다. 현 번들에는 DART 접수번호가 없어 원문 링크는 "미연결"로 투명하게 표시한다.

### Phase 3 — 건설 모듈과 감사 워크플로우 (4주) · 기반 정비 중

**목표:** 프로젝트 산업의 복합적인 수익인식·충당 리스크를 설명 가능한 워크플로우로 구현한다.

- 건설 산업 분류·비교군 확정 후 65개사 데이터 적재
- 미청구공사/계약자산, 원가율, 충당부채, 차입금·현금흐름 테마 구현
- “확인 질문”, 사용자 메모, 신호 선택, 리포트 템플릿 제공
- 보고서의 데이터 기준·규칙 버전·제한사항 자동 표기

**비교그룹 원칙:** 현재 건설 65개사는 모두 `UNCLASSIFIED`로 적재한다. 이는 임의의 A/B/C 분류로 벤치마크를 왜곡하지 않기 위한 안전장치다. A/B/C는 시가총액·사업 포트폴리오 같은 외부 대리변수로 자동 부여하지 않고, 아래 업무 기준을 승인받아 확정한다.

| 그룹 | 승인 전 정의 초안 | 필수 확인값 |
| --- | --- | --- |
| A | 대형 종합건설·복합개발, 장기 프로젝트 비중이 높은 비교군 | 사업보고서의 수주·주택·해외/플랜트 구성 |
| B | 중견 종합건설·전문공사 중심 비교군 | 공종·수주 구조·지역/공공 비중 |
| C | 개발·시행·임대 등 비건설 매출 혼합 비교군 | 매출 인식 방식·PF/임대 자산 비중 |

**이번 정비 결과:** 읽기 전용 품질 리포트는 건설 2,974행·65개사·2023–2025년, 기업 마스터 식별자 누락 0건·재무 사실 중복 0건을 확인한다. 산업맵 65건은 업로드 시 `--auto-map-unclassified-industries`로 생성되는 예상 경고이며, A/B/C 분류 누락 자체를 오류로 취급하지 않는다.

**현재 완료 범위 추가:** 산업 랜딩의 건설 카드에서 65개사 전체를 매출액·영업이익률·계약자산/매출·매출채권/매출·차입금/자산·영업현금흐름/매출로 정렬하는 `탐색 비교 보기`를 제공한다. 이 표는 CFS 우선/OFS 보조의 데이터 탐색용이다. 계약자산·매출채권·차입금 비율은 값이 있는 전체 표본의 3사분위 이상, 영업현금흐름/매출은 음수일 때 검토 신호로 표시한다. 이는 정식 동종 벤치마크나 결론이 아니다. 기업 마스터의 `분류`와 `수익인식 코드`를 근거로 A/B/C **후보**와 사유를 보여 주지만, 이는 검토 큐일 뿐 `industry_map`의 공식 그룹을 변경하지 않는다.

**현재 완료 범위 추가:** 건설 비교 행의 `기업 상세 열기`는 `/companies/{corp_code}?industry=construction`으로 이동한다. 매출액·계약자산(미청구공사)·매출채권·차입금·충당부채·영업활동현금흐름의 3개년 흐름, 신호별 감사 확인 질문, 브라우저 검토 메모·링크 복사·인쇄를 제공한다. 비교 기준은 A/B/C 확정 전 전체 표본 기준임을 상세 화면에도 명시한다.

**완료 기준:** 사용자가 건설 기업의 신호를 선택해 검토 메모가 포함된 리포트를 만들고, 다른 사용자가 동일 컨텍스트를 다시 열 수 있다.

- 역할 기반 접근, 감사 로그, 공유/권한 정책

### Phase 5 — 코스피·코스닥 전체 시장 확장 (계획 단계) · 착수 전

**목표:** 3개 산업(방산·반도체·건설, 144개사)에 한정된 현재 커버리지를 코스피·코스닥 전체 상장기업으로 넓혀, 산업 분류와 무관하게 모든 상장사의 기본 재무비율 분석(2026-07-18에 만든 표준 계정 세트)을 가능하게 한다. UI에는 `전체` / `코스피` / `코스닥` 카테고리를 방산·반도체·건설 탭과 나란히 추가하고, 이 3개 카테고리 선택 시에는 산업 특화 지표(계약자산, CAPEX 등) 없이 공통 재무비율 열만 표시한다(현재 비교 표가 이미 공통 9열 + 산업별 추가 열 구조라 이 요구사항 자체는 기존 구조로 충족됨).

**2026-07-18 조사 결과:**

| 항목 | 수치 |
| --- | ---: |
| 코스피 상장사 (Naver 시가총액 기준) | 2,478개 |
| 코스닥 상장사 | 1,821개 |
| 합계 | 4,299개 (현재 144개사 대비 약 30배) |
| DART 호출 예상 (5개년 × CFS/OFS) | 약 43,000건 |

**현실적 제약:**
1. DART Open API는 키당 일일 호출 한도가 있다(통상 2만 건/일 수준). 4.3만 건은 한 번에 못 받고 최소 2~3일에 걸쳐 나눠 받아야 한다.
2. 코스피·코스닥 종목 목록은 DART의 `corpCode.xml`만으로는 시장 구분(KOSPI/KOSDAQ)이 없어, 기존 `scripts/dart/main.py`의 `MarketCapSelector`(Naver 시가총액 페이지 크롤링)로 시장별 종목 목록을 먼저 확보하고 DART corp_code와 매칭해야 한다.
3. 4,000개 이상의 행을 지금 방식(정렬만 되는 단일 테이블, 페이지네이션 없음)으로 그대로 렌더링하면 브라우저 성능 문제가 생길 수 있다 — 가상 스크롤 또는 페이지네이션이 필요하다.
4. `companies_basic`/`industry_map`은 지금 144개사 규모를 가정한 수동 검수 흐름(중복 차단, 건설 미분류 매핑 등)으로 되어 있어, 4,300개사 규모에서도 같은 정합성 검증이 통과하는지 별도 확인이 필요하다.

**단계별 진행안:**
1. 코스피/코스닥 종목 목록 확보 및 DART corp_code 매칭 → `companies_basic`/`industry_map`에 `kospi`/`kosdaq` 멤버십으로 적재 (기존 기업은 산업 소속과 시장 소속을 동시에 가짐 — 데이터 모델이 이미 다중 소속을 지원)
2. 코스피 우선 → 코스닥 순으로 DART 재추출 (일일 호출 한도 내에서 분할 실행)
3. 표준 재무비율 계정 세트(13개)를 `kospi`/`kosdaq` 산업 테이블에 동일하게 적재, `INDUSTRY_COMPARISON_ACCOUNTS`에 등록
4. 프론트엔드: `전체`/`코스피`/`코스닥` 탭 추가, 대량 행 렌더링을 위한 페이지네이션/가상 스크롤 적용
5. 적재 후 `diagnose_data_gaps.py`로 대규모 커버리지 재확인

## 현재 구현과 실행

### 저장소 구조

```text
risk_analyzer/
  frontend/                 React/Vite UI
  backend/                  FastAPI 분석 API
  scripts/dart/             선택 기업 DART 원천 추출기(main.py) + 결측 계정 보강(fill_missing_accounts.py)
  scripts/pipeline/         표준화/서버 DB/동기화 스크립트
  scripts/supabase/         CSV bundle 검증·Supabase 적재기 + 결측 진단(diagnose_data_gaps.py)
  look/                     최신 기획/원본 CSV 참고 번들 (현재 Git 미추적)
  upload/                   현재 배포용 CSV 번들
  docs/                     데이터·계정·디자인 문서
```

### 로컬 실행

백엔드:

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```

프론트엔드:

```bash
cd frontend
npm install
npm run dev
```

환경변수:

```bash
# backend/.env
DATA_SOURCE=supabase
SUPABASE_DATABASE_URL=postgresql://...
FRONTEND_ORIGIN=http://localhost:5173

# frontend/.env
VITE_API_BASE_URL=http://localhost:8000
```

### 기존 데이터 파이프라인

```bash
python3 analyze_accounts.py
python3 normalize_accounts.py
python3 create_server_db.py
backend/.venv/bin/python sync_supabase.py
```

CSV 적재기는 다음 명령으로 사전 검증한다. 승인된 최신 번들은 날짜 보정·미분류 산업맵·종목코드 정합화 옵션을 함께 명시해야 하며, 이 검증은 DB를 변경하지 않는다.

```bash
backend/.venv/bin/python scripts/supabase/upload_csv_bundle.py \
  --upload-dir look \
  --default-updated-at 2026-07-11 \
  --auto-map-unclassified-industries \
  --reconcile-stock-codes \
  --validate-only
```

전체 public 스키마를 교체하는 적재는 파괴적 작업이므로, 품질 리포트 승인 뒤에만 실행한다.

```bash
backend/.venv/bin/python scripts/supabase/upload_csv_bundle.py --drop-all-public-tables
```

## 배포

- Frontend: Cloudflare Pages
- Backend: Render
- Database: Supabase Postgres

운영 환경은 `DATA_SOURCE=supabase`, `SUPABASE_DATABASE_URL`, `FRONTEND_ORIGIN`을 설정한다. 데이터 갱신은 운영 DB에서 직접 수정하지 않고, 검증된 번들·실행 이력·승인 절차를 통해서만 반영하는 것을 목표로 한다.

## 용어와 면책

- **위험 신호:** 추가 검토가 필요한 패턴. 오류·부정·감사의견을 단정하지 않는다.
- **비교군:** 선택한 산업/그룹과 데이터 충족 조건을 통과한 기업 집합.
- **CFS/OFS:** 연결재무제표/별도재무제표 기준.
- **데이터 기준일:** 해당 분석에 쓰인 최신 재무 데이터의 기준일. 업로드 날짜와 구분한다.

ARI의 결과는 전문가의 판단을 돕기 위한 분석 보조 정보이며, 감사 결론·투자 조언·신용 판단을 대체하지 않는다.
